# Improvements

- [x] [functional] Fix _show_status glob pattern: uses `probe_round_*.txt` but files are named `check_round_*.txt`, causing status to always report 0 check results
- [x] [functional] Add `--timeout` flag to README Usage section — it's implemented in code but not documented, and the default 30s is too low for most real test suites
- [x] [functional] The evolve.py docstring shows `evolve <project-dir>` but the actual CLI requires `evolve start <project-dir>` — update docstring to match, and add `status` subcommand documentation to align with README Usage section
- [x] [functional] In agent.py `run_claude_agent`, the conversation log file is opened without a context manager (`log = open(...)`) — if an exception occurs the file handle leaks. Refactor to use `with open(...) as log:` for proper resource cleanup
- [x] [functional] In `loop.py` `_run_party_mode`, the party agent conversation is saved as `conversation_loop_0.md` (round_num=0) which is misleading and could collide with real round numbering — use a distinct filename like `party_conversation.md` to clearly distinguish the party mode session
- [x] [functional] The `_get_current_improvement` function in `loop.py` returns the first unchecked item regardless of `[needs-package]` tag — when `--yolo` is not set, items tagged `[needs-package]` should be skipped by the orchestrator so the agent doesn't get stuck trying to implement a blocked improvement
- [x] [functional] The `evolve_loop` in `loop.py` doesn't pass the `yolo` flag to `_get_current_improvement` for the post-round re-read (line ~176 in `run_single_round`), and the convergence check in the loop doesn't account for the case where all remaining unchecked items are `[needs-package]` — the loop should detect this and stop early with a message instead of running max_rounds with no progress
- [x] [functional] The `prompts/system.md` template uses `{project_dir}`, `{run_dir}`, and `{yolo_note}` placeholders via Python `.format()`, but if the README or any project content contains literal curly braces (e.g. JSON examples, Rust code), the `.format()` call will crash with a `KeyError` — switch to safer string substitution (e.g. `str.replace()` or `Template`) in `agent.py` `build_prompt`
- [x] [functional] The `_show_status` command doesn't report `[needs-package]` blocked improvements — when evolution stalls because all remaining items are blocked, the operator has no way to see this from `evolve status` output. Add a blocked count (e.g. "2 blocked (needs-package)") using the existing `_count_blocked` helper from `loop.py`
- [x] [functional] The `_count_blocked` function in `loop.py` uses a naive `"[needs-package]" in text` check which causes false positives when an improvement's description merely *mentions* `[needs-package]` (e.g. inside backticks) — refine the check to only match `[needs-package]` as a tag token (e.g. right after the type tag like `[functional] [needs-package]`) rather than anywhere in the description text
- [x] [functional] The `agent.py` `build_prompt` function duplicates the `[needs-package]` tag-matching regex inline (`re.match(r"\[[\w-]+\]\s+\[needs-package\]", text)`) instead of using the shared `_is_needs_package` helper from `loop.py` — import and reuse the helper to keep the logic DRY and prevent future divergence if the pattern changes
- [x] [functional] The README describes a modern TUI powered by `rich` with colored panels, progress bars, real-time agent activity feed, and graceful fallback to plain text — but the codebase uses only plain `print()` statements with no `rich` integration at all. Implement the TUI module with `rich` panels/progress bars and fallback to plain text when `rich` is not installed
- [x] [functional] The `run_claude_agent` function in `agent.py` uses a bare `stream.__aiter__()` / `ait.__anext__()` loop pattern instead of the cleaner `async for message in stream:` idiom — and more critically, it doesn't handle the case where the agent writes partial/streaming tool inputs (the `include_partial_messages=True` option is set but partial messages are not filtered), which could log incomplete tool calls. Refactor to use `async for` and skip partial messages properly
- [x] [functional] The `analyze_and_fix` function in `agent.py` duplicates the retry logic for `RuntimeError` and generic `Exception` — both branches perform identical rate-limit detection and backoff. Consolidate into a single `except Exception` handler, and extract the retry/backoff logic into a helper to reduce code duplication and make it easier to adjust retry behavior
- [x] [functional] The `_run_party_mode` function in `loop.py` has a hardcoded benign-RuntimeError check (`"cancel scope" not in str(e)`) and no rate-limit retry logic at all — if the SDK hits a rate limit during party mode it fails immediately. Reuse the new `_is_benign_runtime_error` and `_should_retry_rate_limit` helpers from `agent.py` for consistency, and add retry support so party mode is as resilient as `analyze_and_fix`
- [x] [functional] The README documents `--model` flag and `EVOLVE_MODEL` env var for selecting the Claude model, and `--resume` flag for resuming interrupted sessions, but neither is implemented in the CLI — add both flags to `evolve.py` arg parsing, pass the model through to `agent.py`, and implement resume logic in `loop.py` to detect last completed round and continue
- [x] [functional] The README specifies exit codes (0=converged, 1=max rounds/remaining, 2=error) and a `TUIProtocol` enforcing method parity between RichTUI and PlainTUI at type-check time, and a `tests/` directory with `test_loop.py`, `test_agent.py`, `test_tui.py`, `test_evolve.py` — implement the Protocol class in `tui.py` and create the test suite
- [x] [functional] The README describes `.evolverc` or `evolve.toml` configuration file support under "Future directions" — while these are aspirational, the README's "Development" section says `pytest tests/ --cov` should work, but no `pytest.ini` or `pyproject.toml` configures the test root or coverage settings — add a `pyproject.toml` with `[tool.pytest.ini_options]` and `[tool.coverage]` sections so `pytest tests/ --cov=. --cov-report=term-missing` works cleanly out of the box
- [ ] [performance] The test coverage is only 37% overall — add targeted tests for the untested pure-logic paths in `loop.py` (e.g. `_count_improvements`, `_count_blocked`, `_detect_last_round` resume logic) and `evolve.py` (e.g. `_show_status` output formatting) to improve coverage above 60% without requiring SDK mocks
